## 概要

`for (i = 0; i < n; i++) { a[i] = a[i] * 2; }`のようなループは、本体の処理(`a[i] = a[i] * 2`)が単純であればあるほど、ループ制御そのもの(条件`i < n`の判定、カウンタ`i`の更新、次の反復へのジャンプ)にかかる相対的なオーバーヘッドが目立つようになる。ループアンローリングは、ループ本体を複数回分(例えば4回分)まとめて複製し、ループの反復回数をその分だけ減らすことで、このオーバーヘッドの発生回数そのものを削減する。単純な変換でありながら、現代のCPUが持つ命令レベル並列性(複数の命令を同時にパイプライン処理する能力)をコンパイラがより有効に引き出せるようになるという実質的な高速化効果を持つ、伝統的かつ実用的な最適化技法である。

## 仕組み

1. ループの反復回数`n`と展開係数`k`(何回分をまとめるか、4や8がよく使われる)を決める
2. ループ本体を`k`回分連続して複製し、カウンタの更新も1回にまとめる(`for (i = 0; i < n; i += 4) { a[i]=a[i]*2; a[i+1]=a[i+1]*2; a[i+2]=a[i+2]*2; a[i+3]=a[i+3]*2; }`のような形になる)
3. これにより、条件判定とカウンタ更新にかかる命令が、元のループの`1/k`の回数で済むようになる
4. 反復回数`n`が展開係数`k`で割り切れるとは限らないため、割り切れない残りの反復を処理する「余り処理(remainder loop)」——展開されていない元の形のループを、割り切れなかった分だけ実行する小さなループ——を末尾に追加する
5. コンパイラは、`n`がコンパイル時に定数として分かっている場合はループ全体を完全に展開してしまう(余り処理も不要になる)こともあれば、`n`が実行時にしか分からない場合は部分的な展開+余り処理という形を取ることもある

## 特性・トレードオフ

- **コンパイル時のコストは小さいが実行時の効果は文脈依存**: 変換自体はコード生成時に機械的に行えるため軽量だが、実際の高速化効果はループ本体の性質(単純な計算か、複雑な分岐を含むか)やCPUのキャッシュ・パイプラインの特性に強く依存する——展開しすぎるとコードサイズが増えて命令キャッシュのヒット率が下がり、かえって遅くなることもある
- **命令レベル並列性を引き出しやすくするという本質的な狙い**: 展開後の複数のループ本体のコピーは、互いにデータ依存関係がなければ(異なる配列要素への独立した操作であれば)CPUが並行してパイプライン処理しやすくなる——単にループ制御命令を減らすだけでなく、後続の命令スケジューリング最適化(命令の並べ替えによるパイプラインストールの削減)の効果を引き出す土台にもなる
- **ベクトル化(SIMD命令の活用)との相性**: ループアンローリングによって明示的に並んだ複数の同種の演算は、SIMD命令(1つの命令で複数のデータに同じ演算を同時に適用する)へのベクトル化のパターンとしてコンパイラに認識されやすくなる——現代のコンパイラの自動ベクトル化パスは、しばしばループアンローリングと組み合わせて実装されている
- **展開しすぎることの弊害**: 展開係数`k`を大きくしすぎるとコードサイズが膨れ上がり、命令キャッシュに収まらなくなって逆に性能が悪化する「コードブロート」を招くことがある——最適な展開係数はターゲットCPUのキャッシュサイズやパイプライン構造に依存するため、コンパイラは通常ヒューリスティックや実測データに基づいて判断する
- **使いどころ**: 数値計算・画像処理のような単純な演算を大量に繰り返すホットループの最適化、組み込みシステムにおける手動でのパフォーマンスチューニング、GPU・SIMDプロセッサ向けのコード生成における自動ベクトル化の前段階
