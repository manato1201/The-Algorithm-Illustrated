# UIデザイン仕様書 — 状態分離型 インタラクティブ・アルゴリズム図鑑

## 1. 前提・スコープ

アーキテクチャ(Web Workers / D3.js・Canvas API / IndexedDB / Vercel Edge Functions BFF / diffベース状態記録)は決定済みであり、本書のスコープ外。本書は見た目・UX・デザインシステムのみを固める。

- **モード**: ハイブリッド(4.5節参照) — ナビゲーション・説明・操作性は実務モード品質を死守しつつ、状態遷移の可視化(アルゴリズム実行体験)だけショーケースティアの演出を許可する
- **デザイン方向**: ダーク×発光(`web-production-skill`の`design-directions.md`の8方向より選定。テック/データビジュアライゼーションとの親和性、WebGL適性◎)
- **フレームワーク**: Next.js (App Router + TypeScript)、Vercelデプロイ、Edge FunctionsをBFFとして使用
- **UI言語**: 日本語のみ(i18nは将来拡張として保留)

`web-production-skill`のうち本プロジェクトに適用したもの: `design-directions.md`(8方向+反復防止のdesign-log運用)、`layout-patterns.md`(7ヒーロー型+セクション装置+構成ルール)、`coding-standards.md`(CSSカスタムプロパティのトークン層構造)、`encoding-and-quality.md`(文字化けゼロ・画質劣化ゼロ・「AI感」チェックリスト)。非該当と判断したものは4.4節末尾に明記する。

> **2026-07-08追記(参考画像による具体化)**: ユーザーから「サイバー系HUD/ターミナル」トーンの参考画像(黒背景+amber系の数値・罫線UI、コーナーの罫線装飾、破線パネル、等幅フォントの統計表示、ステータスチップ+ライブ時計、タイムスタンプ付きログフィード)が共有された。これは「ダーク×発光」方向の具体的なサブスタイルとして採用し、2.1節(デュアルアクセント)と2.6節(HUD装飾要素・全画面共通)に反映した。

## 2. デザイントークン

### 2.1 コアカラーパレット(6色)

| 名前 | 変数名 | Hex | 用途 |
|---|---|---|---|
| Void(地の黒) | `--color-bg-void` | `#06070A` | 全体背景。純黒`#000`を避け僅かに青みを足し平板さを防ぐ |
| Surface(面) | `--color-bg-surface` | `#12141B` | カード・パネル・ヘッダー |
| Surface-2(入れ子面) | `--color-bg-surface-2` | `#191C26` | コードブロック・統計チップ等、一段浮いた面 |
| Text(本文) | `--color-text` | `#EDF0F5` | 純白`#FFF`を避けグレアを抑制 |
| Muted(補助) | `--color-text-muted` | `#8B93A7` | キャプション・補足・非アクティブラベル |
| **Accent-Green(状態/成功シグナル)** | `--color-accent-green` | `#4DFFB0`(phosphor green) | システムステータス(オンライン・完了・確定)を示す成功シグナル専用 |
| **Accent-Amber(UIクロームの主色)** | `--color-accent-amber` | `#FFA733`(HUD amber) | セクションラベル・数値表示・プログレスバー・パネル罫線など、UI装飾全般の主色 |

> **判断根拠**: 当初はphosphor greenの単色運用としていたが、参考画像(サイバー系HUD)を受けてデュアルアクセントに変更。**用途分離ルール**: amber = 「情報・進行中・操作対象」(データ表示、進捗、罫線装飾など画面のほぼ全域)、green = 「状態・成立・成功」(オンライン表示、完了・確定通知など限定的な意味づけ)。この分離により色の氾濫を防ぐ。`encoding-and-quality.md`のAI感チェックリストで紫→青グラデーションが代表的なAI感の指標として挙げられているため、どちらの色域もそれを避けている。

### 2.2 アルゴリズム状態パレット(可視化専用・本プロジェクトのシグネチャー要素)

| 状態 | 変数名 | Hex | 意味 |
|---|---|---|---|
| 未処理 | `--state-idle` | `#3A3F4D` | 未着手の要素 |
| 比較中 | `--state-comparing` | `#FFD23F` | 注目・検査中(amber) |
| 交換中 | `--state-swapping` | `#FF5470` | 変更発生中(coral) |
| 基準/ポインタ | `--state-pivot` | `#7DD3FF` | 軸要素・探索の現在位置(electric blue) |
| 確定/ソート済み | `--state-settled` | `#4DFFB0` | `--color-accent-green`と同色を再利用し「ゴール到達」を一貫させる |

### 2.3 タイポグラフィ

| 用途 | 変数名 | フォント | 備考 |
|---|---|---|---|
| 見出し(和文) | `--font-display` | `"Zen Kaku Gothic New", "Hiragino Kaku Gothic ProN", sans-serif` | 方向の「幾何学ゴシック」要件 |
| 見出し補助(英数字/Big-O表記) | `--font-display-en` | `"Space Grotesk", sans-serif` | 複雑度表記・統計数字用、和文見出しと混植 |
| 本文 | `--font-body` | `"Noto Sans JP", "Hiragino Kaku Gothic ProN", "Yu Gothic", sans-serif` | 可読性優先 |
| コード・データ・状態値 | `--font-mono` | `"JetBrains Mono", "Source Han Code JP", monospace` | 方向の「等幅混ぜ」要件。ゼロにスラッシュがあり誤読防止 |

### 2.4 質感・罫線言語

- **発光罫線(amber)**: `box-shadow: 0 0 12px rgba(255,167,51,0.35)` をアクティブ/フォーカス状態のパネル境界に使用。常時は使わず、静的パネルは非発光の細罫線 `--color-line: rgba(255,167,51,0.14)` のみとし実務モードの落ち着きを保つ
- **発光罫線(green・成功専用)**: `box-shadow: 0 0 12px rgba(77,255,176,0.35)` は「完了・確定・オンライン」を示す瞬間のみに限定使用(2.1節の用途分離ルール)
- **グレイン**: 背景に4〜6%不透明度のノイズテクスチャを疑似要素で重ねる
- **フルブリード**: アルゴリズム実行可視化キャンバスのみ余白ゼロの全幅処理。他UI(説明文・一覧・比較表)はコンテナ幅を維持——ここが唯一の「破調」区画(4節と対応)

### 2.5 モーション言語

- **「点灯」**: `idle`→各状態色への遷移は`opacity`/`filter`(box-shadow相当のglow)のみで行い、transform/opacity原則を厳守
- **パーティクル/グロー・パルス**: 可視化キャンバスの「実行」体験にのみ許可
- **停止ボタンポリシー**: スキル方針(デフォルト全員フル演出、reduced-motion環境の閲覧者にのみ「アニメーションを停止」ボタンを表示)を採用。ただし本プロジェクト固有の拡張として: 状態遷移モーションは装飾ではなく情報(何が起きたか)を伝えるため、停止時も情報を失わない静的代替(例: 一瞬のテキストバッジ「交換」を即時表示)を用意する

### 2.6 HUD/ターミナル装飾要素(全画面共通)

参考画像(サイバー系HUDダッシュボード)から抽出した、**全画面に適用する**構造的な視覚語彙。3節以降の全ビューがこの「共通フレーム」を共有し、サイト全体が1つのシステムダッシュボードとして一貫する。ただし、これは構造・チロームの話であり、モーション/発光の強度は引き続き5節のハイブリッドモード区分に従う(HUDの見た目は全画面共通でも、動きの派手さはコア画面に限定)。

- **コーナーブラケット**: 画面(または主要パネル)四隅にL字型の罫線装飾。`::before`/`::after`疑似要素で`border-top`+`border-left`(他3隅も同様)、amber系1px線、8〜16px角
- **破線パネル**: 未実行・生成中・プレースホルダ領域は`border: 1px dashed rgba(255,167,51,0.4)`。実データが入ると実線罫線に切り替わる(状態の可視化を罫線自体に持たせる)
- **等幅数値表示**: 主要な数値(件数・%・タイマー・ステップ数)は`--font-mono`+`letter-spacing`広めで統一。ラベルは小型正方形ビュレット「■」+トラッキング広めの英字/和文混植ラベル(例: `■ RESEARCH バズ走査`相当の書式)
- **ステータスチップ+ライブ時計**: ヘッダー右上に常時表示する共通コンポーネント。ドット+状態文字列(例: `● SYSTEM ONLINE`)は2.1節の用途分離ルール通りgreenを使用、隣に等幅フォントのライブ時計。本プロジェクトでは「アイドル/実行中/完了」等アプリの実行ステータス表示に転用する
- **ログフィード**: タイムスタンプ+ソースタグ+短い説明を縦積みで流す帯。詳細画面では「アルゴリズム実行のステップイベント履歴」、カタログ/更新情報画面では「データ取得・更新履歴」に転用する共通パターン
- **プログレスバー**: ラベル+塗りつぶしバー+右端%数値、amber系で統一(HOOK TYPES相当の使い方は、本プロジェクトではフィルタ内訳・複雑度分布などの表示に転用可)
- **ステップ式ナビ**: ヘッダー中央の「01/02/03/04」のような番号付きナビラベルは、将来コマンドパレット等を検討する際の参考パターンとして記録するのみ(3節の画面一覧とは1:1対応させない。無理に模倣しない)

## 3. 画面/ビュー一覧

全画面共通で2.6節のHUDフレーム(ステータスチップ+ライブ時計・コーナーブラケット・等幅数値表示)を適用する。以下の「シグネチャー要素」列は、それに加えて各画面固有で許可するモーション/発光演出を示す。

| # | ビュー | 役割 | シグネチャー要素 |
|---|---|---|---|
| 1 | ホーム/カタログ | カテゴリ別アルゴリズム一覧、検索・フィルタ、複雑度バッジ(AST解析結果) | 導入部のみ軽い点灯モーション |
| 2 | **アルゴリズム詳細(コア画面)** | 説明パネル(存在理由・要点・特性/トレードオフ)+インタラクティブ可視化キャンバス+タイムトラベル・スクラバー(再生/一時停止/ステップ/巻き戻し) | **主戦場**。フルブリード可視化+発光状態遷移+パーティクル |
| 3 | 比較ビュー | 複数アルゴリズムの特性を並べて比較(速度競争ではなくトレードオフ表中心) | 各キャンバスの状態遷移のみ、他は実務モード |
| 4 | 更新情報(RSSフィード) | Edge Functions BFF経由で外部記事を取得しカード表示 | なし(実務モード) |
| 5 | About | 個人開発の背景・技術スタック紹介(優先度低) | なし |

> 補足: コマンドパレット風検索(Cmd/Ctrl+K)はテック層向けかつダーク×発光と親和性が高いが、**本セッションでは未決定**のアイデアとして記載するのみ。

## 4. レイアウト原則の転用/非転用

### 4.1 転用する原則(`layout-patterns.md`より)

- **リズム変化**: カタログ(余白多め)→詳細画面(情報密度高)→About/フッター(余白多め)
- **破調1箇所**: 詳細画面のフルブリード可視化キャンバス
- **同一装置の連続回避**: カタログの全カードを均一にせず、代表アルゴリズム1〜2件を大きいショーケースタイル、残りを密なリストに
- **ヒーロー型の転用**: 7型のうち「1. タイポグラフィック全面」をカタログ導入部の"開始の瞬間"として再解釈(マーケティングヒーローではなくアプリ入口の演出)
- **セクション装置の転用**:
  - 「スティッキーサイド」→ 詳細画面(左固定=可視化+コントロール、右スクロール=説明文)
  - 「数字スタッツ」→ 実行中のライブ統計(比較回数・交換回数・ステップ数)
  - 「タイムライン」→ 装飾でなく機能そのもの(スクラバー)
  - 「FAQアコーディオン」→「この手法を避けるべき場面」等の特性/トレードオフ折り畳みに転用

### 4.2 非転用として明記するもの

`forms.md`(問い合わせフォーム不要)、`conversion-design.md`のFV3/CTA/信頼要素/料金表(コンバージョン設計は目的外)、`hearing-guide.md`(クライアントヒアリング前提)、`cms-and-updates.md`(クライアント更新運用、非該当)、`wordpress-theme.md`(Next.js/Vercel運用のため非該当)、`renewal.md`(既存サイト移行なし)、`multipage-guide.md`の納品前提の下層ページ運用ルール(URL設計の考え方のみ軽く参考)。

## 5. ハイブリッドモードの適用範囲

HUDの構造的な見た目(コーナーブラケット・破線パネル・等幅数値・ステータスチップ・ログフィード=2.6節)は全画面共通で、この節の対象外。ここで区分するのは**モーション/発光の強度**のみ。

| 領域 | モード | 理由 |
|---|---|---|
| ナビゲーション、検索/フィルタ、比較表、説明文パネル、コントロールUI | 実務モード(厳格) | 学習ツールとして迷わない・読める・速いことが最優先 |
| アルゴリズム実行の可視化キャンバス、状態遷移(点灯/パーティクル/グロー) | ショーケースティア | このプロジェクト唯一の「見せ場」=存在理由そのもの |

## 6. 技術基盤とトークンの対応(将来のコードへの落とし込み方針)

今回はスキャフォールディングしないが、方針のみ明記する。

- `app/globals.css`の`:root` — CSSカスタムプロパティとして2節の値をそのまま定義(CSS Modules等からのスタイリングに使用)
- `lib/design-tokens.ts` — 同じ値をTS定数としてエクスポート。Canvas 2D描画・D3のカラースケール・Web WorkerへのpostMessage等、CSS変数を直接読めない文脈で使用
- 両者は当面手動同期(プロジェクト規模的にトークン生成スクリプトは過剰。将来必要ならJSON→両方生成の拡張は容易)
- Tailwind導入の是非は本セッションでは未決定のまま据え置き(採用時も同じhex値を移すだけなので急がない)
